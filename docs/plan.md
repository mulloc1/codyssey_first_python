# Quiz Game 구현 작업 계획

## 문서 목적

- `subject.md` 요구사항(필수 + 보너스)을 실제 구현 가능한 작업 단위로 분해한다.
- 각 작업 단위마다 완료 기준을 명확히 해서, 개발/검증/커밋 흐름을 일관되게 유지한다.

## 작업 단위 분해 원칙

- 실행 가능한 작은 단위로 나눈다. (한 번에 한 기능)
- 각 단위는 "코드 구현 + 동작 확인 + 커밋"을 한 세트로 본다.
- 데이터 모델/저장/입력 예외처럼 여러 기능에 공통되는 요소를 먼저 만든다.
- 브랜치 작업이 필요한 기능(퀴즈 풀기)은 별도 기능 브랜치에서 구현 후 `--no-ff`로 병합한다.
- 각 단위는 가능한 한 "독립 테스트 가능한 모듈 경계"로 나눈다.
- 모듈 단위 테스트가 어려운 UI 루프 로직은 입력 처리 함수와 비즈니스 로직을 분리해 검증 가능성을 확보한다.

## 모듈 단위 구분 기준 (테스트 가능성 중심)

- 입력 검증 모듈: 주어진 문자열 입력을 정수/범위 값으로 안전하게 변환하는지 독립 검증 가능해야 한다.
- 도메인 모델 모듈(`Quiz`): 생성/정답 판정/직렬화가 I/O 없이 메모리 상태만으로 검증 가능해야 한다.
- 상태 저장 모듈(`state.json` 로드/저장): 파일 없음/손상/정상 케이스를 분리해 검증 가능해야 한다.
- 게임 로직 모듈(`QuizGame` 핵심 메서드): 메뉴 I/O와 점수 계산/퀴즈 선택 로직을 분리해 핵심 계산을 독립 검증 가능해야 한다.
- 보너스 모듈(랜덤/힌트/히스토리/삭제): 기능별 입력-출력이 분리되어 개별 검증 가능해야 한다.
- 통합 모듈(`main.py`): 예외 종료 포함 전체 흐름이 수동 시나리오로 검증 가능해야 한다.

## 작업 단위 목록

### 0) 프로젝트 초기화

**목표**

- 실행 시작점과 기본 파일 구조를 만든다.

**작업**

- `main.py`, `quiz.py`, `quiz_game.py`, `README.md`, `.gitignore` 생성
- 기본 실행 진입 로직(앱 시작/종료 메시지 수준) 구성

**완료 기준**

- `python main.py` 실행 시 프로그램이 시작되고 종료된다.

---

### 1) 공통 입력/예외 처리 기반

**목표**

- 메뉴/정답 입력 등 모든 숫자 입력에서 동일한 검증 규칙을 적용한다.

**작업**

- 입력 앞뒤 공백 제거 (`strip()`)
- 빈 입력 (Enter만) → 안내 메시지 출력 후 재입력 흐름으로 복귀
- 숫자 변환 실패 (예: `abc`) → 안내 메시지 출력 후 재입력 흐름으로 복귀
- 허용 범위 밖 숫자 (예: 메뉴 `9`, 정답 `0`) → 안내 메시지 출력 후 재입력 흐름으로 복귀
- `KeyboardInterrupt` → 안내 메시지 출력 후 저장 및 안전 종료
- `EOFError` → 안내 메시지 출력 후 저장 및 안전 종료
- 파일 없음/손상 예외는 Step 5에서 처리한다.

**완료 기준**

- [재입력 케이스] 빈 입력 / 숫자 변환 실패 / 범위 초과 입력 시 안내 메시지가 출력되고 입력 프롬프트가 다시 나타난다.
- [종료 케이스] `Ctrl+C` / `EOF` 발생 시 안내 메시지가 출력되고 재입력 없이 프로그램이 안전하게 종료된다.

---

### 2) `Quiz` 클래스 구현

**목표**

- 문제 1개를 표현하는 객체 모델을 완성한다.

**작업**

- 속성: `question`, `choices(4개)`, `answer(1~4)`, `hint(보너스 대비)`
- 메서드: 문제 출력, 정답 판정, 직렬화(`to_dict`), 역직렬화(`from_dict`)

**완료 기준**

- `Quiz` 인스턴스 생성/출력/정답 확인/딕셔너리 변환이 정상 동작한다.

---

### 3) 기본 퀴즈 데이터 구성

**목표**

- 주제 기반 퀴즈를 최소 5개 이상 확보한다.

**작업**

- 문제/선택지4개/정답/힌트 포함 데이터 작성
- 기본 데이터 생성 함수 준비

**완료 기준**

- 초기 실행 시 최소 5개 퀴즈가 메모리에 준비된다.

---

### 4) `QuizGame` 클래스 골격 + 메뉴 루프

**목표**

- 게임 전체 제어 구조를 만들고 메뉴 이동을 가능하게 한다.

**작업**

- 속성: `quizzes`, `best_score`, `history`
- 메서드 틀: 메뉴 출력, 선택 분기, 종료
- 잘못된 메뉴 입력 처리

**완료 기준**

- 메뉴가 반복 출력되고, 각 메뉴 번호가 해당 메서드로 이동한다.

---

### 5) 파일 저장/불러오기 (`state.json`)

**목표**

- 재실행 시 데이터가 유지되도록 영속성을 완성한다.

**작업**

- `state.json` 로드/저장 구현 (UTF-8)
- 파일 없음: 기본 데이터 사용
- 파일 손상/입출력 오류: 안내 후 복구(초기화) 처리

**완료 기준**

- 퀴즈 추가 또는 점수 갱신 후 재실행해도 데이터가 유지된다.

---

### 6) 퀴즈 풀기 기능 (브랜치 필수 단위)

**목표**

- 실제 게임 플레이 흐름을 완성한다.

**작업**

- 기능 브랜치 생성: 예) `feature/play-quiz`
- 문제 출제/정답 입력/정오답 피드백/최종 결과 출력
- 퀴즈 없음 케이스 처리

**완료 기준**

- 한 라운드 플레이가 정상 종료되고 점수가 계산된다.
- 플레이 종료 직후 `best_score`가 `state.json`에 저장되고, 재실행 후에도 갱신된 점수가 유지된다.

**브랜치/병합 기준**

- `feature/play-quiz`에서 작업 완료 후 `main`으로 `--no-ff` 병합

**작업**

**완료 기준**

- 추가한 퀴즈가 재실행 후에도 유지된다.
- Git: 퀴즈 추가 기능 완성 후 커밋한다.

### 11) Git 제출 요건 마감

**목표**

- 과제의 Git/GitHub 체크 항목을 모두 충족한다.

**작업**

- 의미 있는 커밋 10개 이상 유지
- 브랜치 생성/병합 기록 최소 1회 이상 확인
- clone/pull 실습 절차:
  1. 미션 저장소를 별도 로컬 디렉터리에 `clone`한다.
  2. 복제된 저장소에서 간단한 변경(예: README에 한 줄 추가) 후 `commit → push`한다.
  3. 기존 로컬 작업 디렉터리에서 `pull`로 변경사항을 가져온다.
  4. `git log` 또는 파일 내용으로 pull 반영 여부를 직접 확인한다.

**완료 기준**

- `git log --oneline`에서 의미 있는 커밋 10개 이상을 확인할 수 있다.
- `git log --all --oneline --graph`에서 브랜치 생성/병합 기록을 확인할 수 있다.
- clone한 디렉터리와 기존 작업 디렉터리 양쪽에서 pull 반영 내용이 일치함을 확인할 수 있다.

## 체크포인트

- 매 작업 단위 종료 시 수동 실행 테스트를 먼저 한다.
- 저장/복구/예외 흐름은 기능 추가 때마다 회귀 테스트한다.
- 커밋 메시지는 "무엇을 왜 바꿨는지"가 드러나게 작성한다.
- 작업 단위 완료 판단 시 "해당 모듈만 떼어 테스트 가능한가?"를 항상 체크한다.

## 모듈별 최소 테스트 체크리스트

- 입력 검증: `""`, `"abc"`, `" 2 "`, `"9"` 같은 경계 입력에서 기대 메시지/재입력 흐름이 유지되는지 확인
- `Quiz` 모델: `is_correct` 정답/오답 케이스와 `to_dict` -> `from_dict` 왕복 변환 일치 확인
- 저장/로드: 파일 없음, 정상 JSON, 손상 JSON에서 각각 복구 정책이 의도대로 동작하는지 확인
- 퀴즈 플레이: 퀴즈 없음/정상 라운드/중간 예외 종료(`Ctrl+C`, `EOF`) 시 저장 및 종료 동작 확인
- 보너스 기능: 랜덤 출제, 문제 수 제한, 힌트 감점, 삭제 반영, 히스토리 누적이 서로 충돌 없이 동작하는지 확인
