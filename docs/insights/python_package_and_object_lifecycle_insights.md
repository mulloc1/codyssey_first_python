# Python 패키지와 객체 생성 인사이트

이 문서는 Quiz Game 프로젝트를 진행하며 정리한 패키지 구조와 객체 생성 흐름 관련 인사이트를 기록한다.

## 1) `__init__.py`를 두는 이유

`__init__.py`는 "이 디렉터리를 패키지로 어떻게 보여줄지"를 정의하는 파일이다.  
실무에서 중요한 역할은 다음 4가지다.

### 1-1. 패키지 경계 명시

- `src/models/__init__.py`가 있으면 `models`를 명시적인 패키지로 다룰 수 있다.
- 팀원이 코드를 읽을 때도 "여기는 모듈 집합의 진입점"이라는 의도가 분명해진다.

### 1-2. 외부 공개 API 정리 (re-export)

- 내부 파일 구조를 외부 코드가 직접 알 필요 없게 만들 수 있다.
- 예를 들어 `quiz_game.py`가 어디에 있든, 외부는 아래처럼 가져오면 된다.

```python
from src.models import QuizGame
```

- 이를 위해 `src/models/__init__.py`에서 명시적으로 재노출한다.

```python
from .quiz import Quiz
from .quiz_game import QuizGame

__all__ = ["Quiz", "QuizGame"]
```

### 1-3. 리팩터링 내성 향상

- 내부 파일명/구조가 바뀌어도 `__init__.py`만 조정하면 외부 import 경로를 유지할 수 있다.
- 즉, 변경 영향도를 패키지 내부로 가둘 수 있다.

### 1-4. 자동완성/정적 분석 친화성

- IDE, 타입체커, 린터가 패키지 경로를 해석하기 쉬워진다.
- 대규모로 갈수록 "명시적 경계"가 유지보수 비용을 줄여준다.

## 2) `__new__`와 `__init__`의 관계

객체 생성은 내부적으로 2단계로 진행된다.

1. `__new__`: 인스턴스를 "만들어 반환"
2. `__init__`: 반환된 인스턴스를 "초기화"

즉 호출 순서는 `__new__` -> `__init__`이다.

### 2-1. 역할 분리

- `__new__`
  - 클래스 인스턴스를 실제로 메모리에 할당해 반환
  - 반환값이 해당 클래스 인스턴스가 아닐 수도 있음 (고급 케이스)
- `__init__`
  - 이미 생성된 인스턴스의 상태를 세팅
  - 반환값을 갖지 않음 (`None`이어야 함)

### 2-2. 일반적인 개발에서의 기준

- 대부분의 애플리케이션 코드에서는 `__init__`만 구현하면 충분하다.
- 검증, 기본값 설정, 의존 객체 연결 같은 로직을 `__init__`에 둔다.
- `__new__` 커스터마이징은 특수 상황에서만 사용한다.
  - immutable 타입 상속
    - `tuple`, `str`, `int` 같은 불변 타입은 값이 만들어진 뒤 수정할 수 없다.
    - 그래서 "초기값을 어떻게 만들지"를 결정하는 시점(생성 단계)인 `__new__`에서 제어해야 한다.
    - 예: `class UserId(str)`에서 공백 제거/형식 정규화를 `__new__`에서 수행
  - 싱글턴/캐싱 객체
    - 같은 인자로 객체를 여러 번 만들 때, 매번 새 인스턴스를 만들지 않고 기존 인스턴스를 재사용하고 싶을 수 있다.
    - 이 경우 `__new__`에서 캐시를 확인하고 기존 객체를 반환할 수 있다.
    - 예: 설정 객체, 연결 풀 핸들러, Flyweight 패턴
  - 인스턴스 생성 제어가 필요한 프레임워크 코드
    - ORM/직렬화/DI 컨테이너 같은 프레임워크는 객체 생성을 내부 규칙으로 통제할 때가 있다.
    - 생성 전에 프록시 객체를 반환하거나, 등록된 메타데이터를 기반으로 다른 클래스를 반환하는 경우가 있다.
    - 이런 "어떤 객체를 반환할지" 결정은 `__init__`가 아니라 `__new__` 단계에서만 가능하다.
